# ContentPulse Query History And Analysis Archive Design

## 背景

当前 ContentPulse 已经具备以下能力：

- 监控分类管理
- 关键词与平台绑定
- 公众号 / 小红书抓取
- SQLite 持久化抓取结果
- 内容、选题分析、监控设置三大 Tab

现状确认如下：

- **已持久化到 SQLite**
  - 关键词目标 `keyword_targets`
  - 抓取批次 `sync_runs`
  - 抓取结果 `collected_contents`
- **未持久化到 SQLite**
  - 搜索历史
  - 选题分析报告快照
  - 选题建议明细
- **当前搜索历史来源**
  - 浏览器 `localStorage`
- **当前选题分析来源**
  - 前端原型数据

当前数据库文件位于 worktree：

- `C:\Users\koubinyue\codex project\Agent内容工厂\.worktrees\content-monitoring-workbench\数据采集与选题分析agent\.codex-data\contentpulse.sqlite`

当前库内已有数据：

- `keyword_targets: 3`
- `sync_runs: 8`
- `collected_contents: 84`

## 目标

在**不破坏现有抓取接口、现有主界面结构、现有内容展示链路**的前提下，新增两类持久化能力：

1. 每次查询的关键词都保存为可回看的历史记录
2. 每次查询对应的选题分析都保存为可回看的历史快照

用户后续应该能够：

- 查看历史查询过哪些关键词
- 查看某次查询抓到了哪些内容
- 查看某次查询对应的选题信息
- 按关键词或按查询批次回看历史
- 一键基于历史查询重新抓取

## 设计原则

### 1. 非侵入式扩展

这轮不推翻现有数据链路，只在现有抓取成功 / 失败流程上挂接历史保存逻辑。

保持不变的部分：

- `/api/content/refresh`
- `/api/content/list`
- 现有监控设置页面结构
- 现有内容 Tab 主区域布局
- 现有搜索历史按钮入口位置

### 2. 现有体验优先

这轮不改成全新页面，而是在现有工作台上增加“历史抽屉 / 历史侧板”的查看能力。

### 3. SQLite 作为主存储

搜索历史和选题历史统一进 SQLite，不再依赖 `localStorage` 作为主来源。

### 4. 兼容当前原型阶段

即使 AI 分析仍为 mock 生成逻辑，也要把“某次查询对应的分析结果”固化成快照，方便后续替换为真实分析服务时不改表结构。

## 需求拆解

### A. 查询历史持久化

每次触发以下动作时，创建一条查询记录：

- 点击内容 Tab 的 `一键更新`
- 新增关键词后自动触发抓取

每条查询记录至少包含：

- 查询关键词
- 所属分类
- 查询平台范围
- 触发方式
- 开始时间 / 结束时间
- 成功 / 失败状态
- 抓取条数信息

### B. 选题分析快照持久化

每次查询结束后，将对应选题分析结果存档。

保存内容包括：

- 今日热点摘要
- 前一天用户关注焦点
- 爆款内容共性拆解
- 洞察建议
- 多条选题建议

### C. 历史查看入口

保留现有右上角 `搜索历史` 按钮，但点击后不再只是轻量弹层，而改为**右侧抽屉式历史面板**。

原因：

- 需要承载历史关键词列表
- 需要承载查询详情
- 需要承载选题详情
- 轻量 popover 空间不够

### D. 历史复用动作

用户在历史面板中应能执行：

- 查看内容
- 查看选题
- 重新抓取

## 数据模型设计

### 1. 新增表：`search_queries`

表示一次真实查询批次。

字段：

- `id`
- `category_id`
- `keyword_target_id`
- `keyword`
- `platform_scope`
- `trigger_type`
- `status`
- `fetched_count`
- `capped_count`
- `started_at`
- `finished_at`
- `error_message`

说明：

- `platform_scope` 允许保存：
  - 单平台：`wechat`
  - 单平台：`xiaohongshu`
  - 多平台：`wechat,xiaohongshu`
- `trigger_type` 允许值：
  - `manual_refresh`
  - `keyword_created`

### 2. 新增表：`analysis_snapshots`

表示某次查询对应的一份分析快照。

字段：

- `id`
- `search_query_id`
- `category_id`
- `keyword`
- `generated_at`
- `hot_summary`
- `focus_summary`
- `pattern_summary`
- `insight_summary`

### 3. 新增表：`analysis_topics`

表示某份分析快照中的具体选题卡。

字段：

- `id`
- `snapshot_id`
- `title`
- `intro`
- `why_now`
- `hook`
- `growth`
- `support_content_ids`

说明：

- `support_content_ids` 用 JSON 数组保存，后续可直接做“查看支撑内容”

## 数据流设计

### 查询写入流程

```text
用户点击一键更新
  -> 创建 search_queries(status=running)
  -> 继续调用现有 refresh 流程
  -> 抓取成功后更新 search_queries(status=success, fetched_count, capped_count)
  -> 保存 analysis_snapshots
  -> 保存 analysis_topics
  -> 页面继续使用现有刷新结果更新
```

### 查询失败流程

```text
用户点击一键更新
  -> 创建 search_queries(status=running)
  -> 调用现有 refresh 流程
  -> 抓取失败
  -> 更新 search_queries(status=failed, error_message)
  -> 页面继续沿用当前已有错误提示逻辑
```

### 历史回看流程

```text
点击搜索历史
  -> 打开右侧历史抽屉
  -> 从 SQLite 读取 search_queries
  -> 默认按 started_at desc 排序
  -> 点“查看内容”后读取这次查询关联的内容快照
  -> 点“查看选题”后读取这次查询关联的分析快照
```

## 4. 历史内容回看

点击 `查看内容` 后，不重新抓取，而是：

- 自动回填分类
- 自动回填关键词
- 自动回填平台
- 自动显示 SQLite 中该次查询对应的内容结果

这样可以做到真正的“历史快照回看”，避免点击历史后又重新走一次实时接口。

## 和现有系统的兼容策略

### 1. 不改现有抓取 API 协议

现有：

- `/api/content/refresh`
- `/api/content/list`

本轮只在服务端内部扩展保存逻辑，不强行改前端现有调用参数结构。

### 2. 不改现有主内容区布局

内容区、分析区、设置区的现有主体结构保持不变。  
新增历史抽屉，不把原页面重新布局。

### 3. 不影响现有 localStorage 搜索历史的短期使用

迁移期间允许：

- SQLite 作为主数据源
- localStorage 作为临时兼容缓存

后续稳定后再决定是否完全移除 localStorage 逻辑。

### 4. 不影响现有选题分析页使用

当前 `选题分析` 页仍然可以继续用现有 mock / 页面态逻辑显示。  
新能力只是把“某次查询对应的分析结果”额外存一份快照。

## 数据库路径建议

当前数据库文件在 worktree 内，容易导致：

- 主目录运行时生成另一份数据库
- 数据被误认为丢失

建议在本轮一并收敛数据库路径到主项目公共目录，例如：

- `C:\Users\koubinyue\codex project\Agent内容工厂\数据采集与选题分析agent\.codex-data\contentpulse.sqlite`

原则：

- 无论主目录还是 worktree 运行，都尽量指向同一份数据库
- 避免数据被分散在多个工作树中

## 验收标准

### 数据持久化

- 每次查询都会在 SQLite 中生成 `search_queries` 记录
- 每次查询对应的分析结果都会生成 `analysis_snapshots` 和 `analysis_topics`
- 历史查询和选题信息在页面刷新后仍可回看

### 页面行为

- 搜索历史入口仍在右上角
- 点击后打开右侧抽屉，不遮挡主操作区
- 用户可以查看历史关键词、历史内容、历史选题
- 点击历史记录不会破坏当前抓取链路

### 兼容性

- 现有抓取接口继续可用
- 现有平台切换、数量统计、一键更新不回退
- 现有内容区、分析区、设置区主结构不被重写

## 风险与边界

### 本轮不做

- 不引入新的 AI 分析服务
- 不把分析结果做成复杂工作流引擎
- 不新增多页路由体系
- 不调整当前主工作台的整体结构

### 已知风险

- 如果继续使用 worktree 私有数据库路径，历史查询会分散
- 如果后续真实 AI 分析字段变化，`analysis_snapshots` 可能需要扩展字段

## 推荐实现顺序

1. 先补数据库 schema
2. 再补 repository 层
3. 在现有 refresh 流程上挂接 `search_queries`
4. 在现有分析数据生成处挂接快照保存
5. 最后补前端历史抽屉与回看逻辑
