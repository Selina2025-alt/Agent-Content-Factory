# ContentPulse 搜索历史页面设计

## 背景

当前 ContentPulse 已经具备以下能力：

- 监控分类管理
- 内容 / 选题分析 / 监控设置三大主视图
- 公众号与小红书关键词抓取
- SQLite 持久化关键词目标、抓取批次与抓取结果

当前“搜索历史”仍存在两个问题：

1. 页面入口和形态不符合产品心智
   - 现在更接近临时抽屉或弹层
   - 不像一个可长期回看的历史页面
2. “历史关键词 + 历史内容 + 历史选题信息”还没有在页面层形成稳定闭环
   - 用户无法像查看档案一样稳定回看
   - 刷新页面或重开项目后，页面层体验还不够明确

本次目标是在不破坏现有抓取接口、现有主界面结构和现有内容链路的前提下，将“搜索历史”升级为一个正式的一级页面视图。

## 目标

本次设计要达成以下结果：

1. 顶部导航新增一级入口 `搜索历史`
2. 用户可以查看之前搜索过的关键词历史
3. 每条历史记录都能关联查看：
   - 当次抓取内容
   - 当次选题信息
4. 刷新页面或重新打开项目后，历史记录仍然存在
5. 数据以 SQLite 为主，不再只是临时显示在前端页面上

## 非目标

本次不做以下改动：

- 不重写现有抓取接口协议
- 不改动现有内容页、分析页、设置页的总体信息架构
- 不引入新的外部数据库
- 不在本次把 AI 分析逻辑替换为真实生成服务

## 当前数据现状

当前已经持久化到 SQLite 的数据：

- `keyword_targets`
- `sync_runs`
- `collected_contents`

当前尚未形成稳定页面闭环的数据：

- 搜索历史页面视图
- 查询历史的页面级列表与筛选体验
- 历史查询对应的选题快照回看体验

## 设计原则

### 1. 不破坏现有稳定能力

保留以下稳定能力不被重写：

- `/api/content/refresh`
- `/api/content/list`
- 当前内容页主视图
- 当前选题分析页主视图
- 当前监控设置页主视图

搜索历史页面是新增视图，不是推翻现有页面。

### 2. 历史页面优先服务“回看”

搜索历史不是二次搜索入口，而是“历史档案入口”。

它需要回答四个问题：

- 我搜过哪些关键词
- 当时抓到了什么内容
- 当时给了什么选题建议
- 现在要不要重新抓一次

### 3. SQLite 作为唯一可信来源

页面中的搜索历史、历史详情、历史选题信息，都优先读取 SQLite。

前端缓存可以保留作为兼容层，但不再作为主要数据来源。

## 信息架构

顶部主导航调整为：

```text
[内容] [选题分析] [搜索历史] [监控设置]
```

说明：

- `内容`：查看当前分类下最新抓取内容
- `选题分析`：查看当前活跃关键词的报告与洞察
- `搜索历史`：查看历史查询记录、历史内容和历史选题
- `监控设置`：管理平台、关键词、账号和规则

## 用户交互设计

### 1. 查看历史记录

流程：

1. 用户点击顶部 `搜索历史`
2. 页面切换到历史视图
3. 默认按查询时间倒序显示
4. 默认展开最近一次查询记录

### 2. 查看历史内容

点击 `查看内容` 后：

- 不重新请求抓取接口
- 直接从 SQLite 读取该次查询对应内容
- 页面切换回 `内容` 视图
- 自动回填：
  - 分类
  - 关键词
  - 平台
  - 内容列表

### 3. 查看历史选题

点击 `查看选题` 后：

- 不重新请求抓取接口
- 直接从 SQLite 读取该次查询对应选题快照
- 页面切换回 `选题分析` 视图
- 自动展示该次查询归档的报告与选题建议

### 4. 重新抓取

点击 `重新抓取` 后：

- 以历史记录中的关键词和平台为准
- 走现有刷新链路
- 成功后生成一条新的历史记录
- 原有历史记录不覆盖，保留快照属性

## 数据模型

### 1. `search_queries`

记录一次真实查询。

字段：

- `id`
- `category_id`
- `keyword_target_id`
- `keyword`
- `platform_scope`
- `trigger_type`
- `status`
- `fetched_count`
- `capped_count`
- `started_at`
- `finished_at`
- `error_message`

页面默认使用口径：

- 列表显示数量：`capped_count`
- 若需要调试再额外展示 `fetched_count`

### 2. `analysis_snapshots`

记录一次查询对应的选题分析摘要。

字段：

- `id`
- `search_query_id`
- `category_id`
- `keyword`
- `generated_at`
- `hot_summary`
- `focus_summary`
- `pattern_summary`
- `insight_summary`

### 3. `analysis_topics`

记录一次选题快照下的具体选题建议。

字段：

- `id`
- `snapshot_id`
- `title`
- `intro`
- `why_now`
- `hook`
- `growth`
- `support_content_ids`

## 数据流

### 查询保存流程

```text
用户点击一键更新
  -> 创建 search_queries(status=running)
  -> 调用现有 refresh 流程
  -> 保存抓取结果到 collected_contents
  -> 更新 search_queries(status=success/failed)
  -> 归档 analysis_snapshots
  -> 归档 analysis_topics
```

### 历史页面读取流程

```text
进入搜索历史页
  -> 读取 search_queries
  -> 按 started_at desc 排序
  -> 读取每条记录对应的内容预览
  -> 若用户选中某条记录
       -> 读取 analysis_snapshots
       -> 读取 analysis_topics
       -> 读取该次查询关联内容
```

## API 设计

保留现有接口不变，新增历史读取接口：

### `GET /api/history`

用途：

- 返回某个分类下的历史查询列表

入参：

- `categoryId`
- `platform`（可选）
- `keyword`（可选）

出参：

- `queries`
- 每条记录可附带预览内容前 3 条

### `GET /api/history/[queryId]`

用途：

- 返回某次查询的完整详情

出参：

- `query`
- `analysis`
- `items`

## 页面状态

### 空状态

当还没有任何历史记录时：

- 显示“还没有搜索历史”
- 引导用户去内容页进行第一次抓取

### 错误状态

当历史读取失败时：

- 显示明确错误提示
- 不覆盖现有内容页状态

### 部分缺失状态

当某条历史记录有抓取结果但没有选题快照时：

- 内容仍可查看
- 选题区域显示“当前查询尚未生成选题快照”

## 兼容策略

### 1. 保留现有抓取逻辑

本次不修改现有平台抓取协议，只在抓取完成后归档。

### 2. 保留现有页面主结构

本次新增的是一级历史页，而不是重写原有内容页。

### 3. localStorage 作为过渡兼容

如果页面仍有旧版 `localStorage` 历史逻辑，可保留作为短期兜底；
但正式历史页读取应以 SQLite 为主。

## 数据库路径建议

当前数据库仍在 worktree 内，长期会导致主目录和 worktree 各自生成数据。

建议下一步将数据库路径收敛到主项目目录，例如：

- `C:\Users\koubinyue\codex project\Agent内容工厂\数据采集与选题分析agent\.codex-data\contentpulse.sqlite`

原则：

- 无论主目录还是 worktree 运行，都尽量指向同一份数据库
- 避免历史数据分叉
